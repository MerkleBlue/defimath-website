4:T2236,<p>Two changes:</p>
<ol>
<li><strong><a href="/docs/math/sqrt/"><code>sqrt</code></a> and <a href="/docs/math/cbrt/"><code>cbrt</code></a> accept the full
<code>uint256</code> domain.</strong> Both errors and their upper-bound constants
(<code>SqrtUpperBoundError</code>, <code>CbrtUpperBoundError</code>, <code>SQRT_UPPER_BOUND</code>,
<code>CBRT_UPPER_BOUND</code>) are removed. Any input in <code>[0, uint256.max]</code> returns a
valid FP18 result.</li>
<li><strong><a href="/docs/math/pow/"><code>pow</code></a> overflow bugfix.</strong> The internal <code>a · ln(x)</code>
multiplication inside <code>unchecked</code> could silently overflow <code>int256</code> for
adversarial exponents (<code>|a| ≳ 3.25 × 10⁵⁶</code>), letting <code>pow</code> return
plausible-looking garbage. New <code>MAX_POW_EXPONENT = 1e54</code> bound with a typed
<code>PowExponentOutOfBoundsError</code> revert — well below the overflow point and
~18 orders of magnitude above any realistic financial exponent
(real value 10³⁶).</li>
</ol>
<p>For a deep dive into the new <code>sqrt</code> — the CLZ seed, the five Newton iterations, and how it lands at 197 gas — see the companion post: <a href="/blog/how-i-wrote-a-fixed-point-solidity-sqrt-that-runs-in-197-gas/">How I wrote a fixed-point Solidity sqrt that runs in 197 gas</a>.</p>
<h2>Gas changes</h2>
<p>Beyond <code>sqrt</code> and <code>cbrt</code>, <code>sqrtTime</code> also picked up the new CLZ-derived seed
algorithm — its savings cascade through every Black-Scholes function that
computes an annualized time term. <code>expm1</code> and <code>log1p</code> shed a few gas each,
which propagates into <code>continuousToDiscrete</code> and <code>discreteToContinuous</code> in
the Rates module. The <code>sqrt</code> savings compound into every statistics function
that calls it — <code>geometricMean</code>, <code>stdDev</code>, <code>historicalVolatility</code>,
<code>sharpeRatio</code>. Options and Binary options both benefit — every function that
depends on <code>sqrtTime</code> (all Greeks, IV, both pricing functions) drops ~15 gas.
<code>pow</code> is the only row that went up: the new <code>MAX_POW_EXPONENT</code> check costs
~40 gas — the price of catching the silent overflow bug.</p>

<h2>API removed</h2>
<ul>
<li><code>SqrtUpperBoundError()</code> and <code>CbrtUpperBoundError()</code> errors deleted.</li>
<li><code>SQRT_UPPER_BOUND</code> and <code>CBRT_UPPER_BOUND</code> constants deleted.</li>
<li><strong>New</strong>: <code>PowExponentOutOfBoundsError()</code> and <code>MAX_POW_EXPONENT</code> for the pow bugfix.</li>
</ul>
<p>Callers who caught <code>SqrtUpperBoundError</code> or <code>CbrtUpperBoundError</code> need to
remove those handlers — those code paths no longer exist. Callers who pass
finite exponents to <code>pow</code> see no behavior change.</p>
