4:T4030,<p>The CLZ opcode landed in EVM Osaka via <a href="https://eips.ethereum.org/EIPS/eip-7939">EIP-7939</a>, exposed in Solidity 0.8.31 as the Yul builtin <code>clz</code>. It costs 3 gas, returns the number of leading zero bits in a 256-bit value, and replaces the loops and lookup tables every previous on-chain library used to compute the same thing. DeFiMath sits on top of it. This post walks through what CLZ actually does, the one-line idiom it enables, and the three places in the library where it's the hot path.</p>
<h2>What CLZ does</h2>
<p>The opcode pops one value off the stack and pushes a single number: how many of that value's 256 bits are zero before its first set bit. A few examples:</p>
<pre><code>clz(0)                 = 256
clz(1)                 = 255
clz(2 ** 80)           = 175
clz(type(uint256).max) = 0
</code></pre>
<p>Two things to note. First, <code>clz(0) = 256</code> — the convention chosen by the EIP. You don't have to special-case zero before calling it, only after. Second, the result is exactly <code>255 − floor(log₂(x))</code> for nonzero <code>x</code>. That equivalence is the entire point: bit length and base-2 logarithm are the same primitive, and now we can read them in 3 gas.</p>
<h2>How to use it in Solidity</h2>
<p>Solidity 0.8.31 exposes <code>clz</code> as a Yul builtin:</p>
<pre class="shiki github-dark" style="background-color:#24292e;color:#e1e4e8" tabindex="0"><code><span class="line"><span style="color:#F97583">function</span><span style="color:#B392F0"> bitLength</span><span style="color:#E1E4E8">(</span><span style="color:#79B8FF">uint256</span><span style="color:#FFAB70"> x</span><span style="color:#E1E4E8">) </span><span style="color:#F97583">internal</span><span style="color:#F97583"> pure</span><span style="color:#F97583"> returns</span><span style="color:#E1E4E8"> (</span><span style="color:#79B8FF">uint256</span><span style="color:#FFAB70"> bits</span><span style="color:#E1E4E8">) {</span></span>
<span class="line"><span style="color:#F97583">    assembly</span><span style="color:#E1E4E8"> { bits </span><span style="color:#F97583">:=</span><span style="color:#B392F0"> sub</span><span style="color:#E1E4E8">(</span><span style="color:#79B8FF">256</span><span style="color:#E1E4E8">, </span><span style="color:#B392F0">clz</span><span style="color:#E1E4E8">(x)) }</span></span>
<span class="line"><span style="color:#E1E4E8">}</span></span></code></pre>
<p>The opcode requires the Osaka EVM target — add this to your <code>hardhat.config.js</code>:</p>
<pre class="shiki github-dark" style="background-color:#24292e;color:#e1e4e8" tabindex="0"><code><span class="line"><span style="color:#B392F0">solidity</span><span style="color:#E1E4E8">: {</span></span>

<p>Targeting an older EVM and calling <code>clz</code> is a <em>compile-time</em> error, not a runtime surprise. Bad config fails fast.</p>
<h2>The bit-length trick</h2>
<p>Every CLZ-based optimization in DeFiMath grows from one identity:</p>
<pre><code>bits = 256 − clz(x)             // bit length of x
floor(log₂(x)) = bits − 1       // for x ≥ 1
</code></pre>
<p><code>bits</code> lets you do two useful things almost for free:</p>
<ol>
<li><strong>Find the largest power of two ≤ x</strong> — <code>2^(bits − 1)</code>. Useful for range reduction.</li>
<li><strong>Find a power of two near the k-th root of x</strong> — <code>2^⌈bits/k⌉</code>. Useful as a Newton-Raphson initial guess.</li>
</ol>
<p>Both compile to one instruction. The rest of the post is what we do with that.</p>
<h2>Use #1: sqrt initial guess</h2>
<p>Newton-Raphson for square root iterates <code>y ← (y + x/y) / 2</code> and roughly doubles the number of correct bits per step. The catch is that you have to <em>start somewhere</em>. A constant initial guess (say, <code>y₀ = 1</code>) needs ~30 iterations to converge to 18 decimals — a non-starter on-chain.</p>
<p>CLZ collapses the initial-guess problem: <code>y₀ = 2^⌈bits/2⌉</code> is within a factor of <code>√2</code> (~1.41) of the true root. From one bit of error, six Newton iterations land bit-exact at the 1e18 fixed-point scale. The <a href="/docs/math/sqrt">DeFiMath sqrt</a> hot path:</p>
<pre class="shiki github-dark" style="background-color:#24292e;color:#e1e4e8" tabindex="0"><code><span class="line"><span style="color:#F97583">assembly</span><span style="color:#E1E4E8"> {</span></span>
<span class="line"><span style="color:#E1E4E8">    x </span><span style="color:#F97583">:=</span><span style="color:#B392F0"> mul</span><span style="color:#E1E4E8">(x, </span><span style="color:#79B8FF">1e18</span><span style="color:#E1E4E8">) </span><span style="color:#6A737D">// scale to 1e36</span></span>
<span class="line"></span>
<span class="line"><span style="color:#6A737D">    // y₀ ≈ 2^⌈bits/2⌉, within factor √2 of sqrt(x)</span></span>

<span class="line"></span>
<span class="line"><span style="color:#E1E4E8">    y </span><span style="color:#F97583">:=</span><span style="color:#B392F0"> shr</span><span style="color:#E1E4E8">(</span><span style="color:#79B8FF">1</span><span style="color:#E1E4E8">, </span><span style="color:#B392F0">add</span><span style="color:#E1E4E8">(y, </span><span style="color:#B392F0">div</span><span style="color:#E1E4E8">(x, y)))  </span><span style="color:#6A737D">// Newton, ×6</span></span>

<span class="line"><span style="color:#E1E4E8">    y </span><span style="color:#F97583">:=</span><span style="color:#B392F0"> shr</span><span style="color:#E1E4E8">(</span><span style="color:#79B8FF">1</span><span style="color:#E1E4E8">, </span><span style="color:#B392F0">add</span><span style="color:#E1E4E8">(y, </span><span style="color:#B392F0">div</span><span style="color:#E1E4E8">(x, y)))</span></span>

<p>That's the whole function. 245 gas average.</p>
<h2>Use #2: cbrt initial guess</h2>
<p>Same trick at a different exponent. For cube root, <code>y₀ = 2^⌈bits/3⌉</code> lands within a factor of <code>∛2</code> (~1.26) of the true cube root. Six iterations of the cube-root Newton update <code>y ← (2y + x/y²) / 3</code> get to bit-exact precision. The <a href="/docs/math/cbrt">cbrt page</a> has the full breakdown — most notably why this lets cbrt be single-branch where sqrt needs an inversion branch for <code>x &#x3C; 1</code>. 368 gas average.</p>
<h2>Use #3: ln range reduction</h2>
<p>The Mercator series for <code>ln</code> converges slowly when <code>x</code> is far from 1. The classic remedy is to reduce <code>x</code> into <code>[1, 2)</code>: find the integer <code>k = floor(log₂(x))</code>, then <code>x / 2^k</code> lives in <code>[1, 2)</code>. The final result is <code>ln(reduced) + k · ln(2)</code>.</p>
<p>Pre-Osaka, computing <code>k</code> required a binary-search-by-shift or a de Bruijn lookup — ~150 gas, dozens of opcodes, the kind of careful thing Solady's <code>log2</code> does. With CLZ it's two instructions:</p>
<pre class="shiki github-dark" style="background-color:#24292e;color:#e1e4e8" tabindex="0"><code><span class="line"><span style="color:#F97583">assembly</span><span style="color:#E1E4E8"> {</span></span>
<span class="line"><span style="color:#F97583">    let</span><span style="color:#E1E4E8"> bits </span><span style="color:#F97583">:=</span><span style="color:#B392F0"> sub</span><span style="color:#E1E4E8">(</span><span style="color:#79B8FF">256</span><span style="color:#E1E4E8">, </span><span style="color:#B392F0">clz</span><span style="color:#E1E4E8">(integerPart))  </span><span style="color:#6A737D">// floor(log₂(integerPart)) + 1</span></span>
<span class="line"><span style="color:#E1E4E8">    x </span><span style="color:#F97583">:=</span><span style="color:#B392F0"> shr</span><span style="color:#E1E4E8">(bits, x)                       </span><span style="color:#6A737D">// reduces x to [1, 2)</span></span>
<span class="line"><span style="color:#E1E4E8">}</span></span></code></pre>
<p><a href="/docs/math/ln">DeFiMath's ln</a> does exactly this, then narrows further to <code>[1, √2)</code> with one more conditional divide so the Mercator series converges in ~10 terms. Total: 375 gas.</p>

<p>Solady is the closest competitor and the only one within 2× on <code>sqrt</code> — they bridge the initial-guess gap with a hand-tuned binary search. The widening gap on <code>cbrt</code> and <code>ln</code> is mostly the savings CLZ gives at the front of each function. PRBMath and ABDK predate the opcode entirely; they pay for log₂ in software at every call.</p>
<p>The two functions where Solady stays competitive are exactly the ones with the cheapest pre-CLZ workarounds. The ones where DeFiMath pulls ahead are the ones that need <em>both</em> a fast initial guess and a fast range reduction — and CLZ delivers both for the price of two opcodes.</p>
<h2>Compounding effects</h2>
<p>The savings don't stop at the primitives. <code>sqrt</code> and <code>ln</code> are the building blocks of Black-Scholes option pricing: both <code>callOptionPrice</code> and <code>putOptionPrice</code> call <code>ln(spot / strike)</code> once and a sqrt-based time-adjusted volatility once per evaluation. Swapping in the CLZ-powered versions of those two primitives drops <a href="/docs/options"><code>callOptionPrice</code></a> from ~3,100 gas to <strong>2,876 gas</strong> — roughly 10% cheaper, with zero change to the option-pricing math itself.</p>
<p>The pattern repeats across the library. Every function that touches a root or a logarithm — IV solving, futures, log returns, historical volatility, Sharpe ratio — gets a free discount the moment its inner primitives switch to CLZ-based versions. The opcode doesn't just save ~150 gas on one <code>log2</code> call; it saves that ~150 gas everywhere <code>log2</code> lives.</p>
<h2>Caveats</h2>
<ul>
<li><strong>Osaka EVM target required.</strong> CLZ is only available on Osaka and later. Targeting Cancun, Shanghai, or earlier won't compile if your code references <code>clz</code>.</li>
<li><strong>Solidity 0.8.31+.</strong> Earlier versions don't expose the Yul builtin. (You can still emit the opcode by hand with <code>verbatim</code>, but at that point you're writing assembly assembly.)</li>
<li><strong><code>clz(0) = 256</code>.</strong> Don't try to guard against zero before the call — the opcode handles it. Guard after, if your algorithm assumes a nonzero input.</li>
</ul>
